# MCP 与 Agent 交互原理简介

本文以本项目为例，说明 **MCP（Model Context Protocol）** 和 **Agent** 分别扮演什么角色、如何配合工作。重点放在「谁提供能力、谁做决策、二者如何交互」，不展开具体实现细节。

---

## 一、先分清两个角色

### MCP：能力的提供方

- **做什么**：把「能做什么」以**工具**的形式暴露出来，并约定好**名字、参数、返回**，供别的程序来调用。
- **不做什么**：不理解用户说了什么，也不决定要不要调用、传什么参数；只负责「收到调用请求就执行，并把结果按约定返回」。

### Agent：决策与编排方

- **做什么**：理解用户意图，决定要调用哪个工具、传什么参数；调用完后把结果再交给大模型，由大模型生成最终回复。
- **不做什么**：一般不自己实现具体业务逻辑，而是依赖外部提供的工具（本地函数或 MCP 工具）。

一句话：**MCP 提供「能做什么」的清单与调用方式，Agent 负责「何时、对谁、用什么参数」去用这些能力。**

---

## 二、交互的整体流程（概念上）

可以把一次「用户说发 hi 到 seatalk」的完整过程，抽象成下面几步（不区分是否经 MCP，先看逻辑）：

1. **用户** 对 Agent 说：「发送 hi 到 seatalk」。
2. **Agent** 把这句话交给 **大模型（LLM）**。
3. **LLM** 理解意图后，决定要调用「发消息」这个工具，并给出参数，例如：`content: "hi"`。
4. **Agent** 拿到「调哪个工具、参数是什么」后，去**执行**这次调用（要么调本地函数，要么按 MCP 协议请求 MCP 服务）。
5. **执行方**（本地代码或 MCP 服务）真正去发 SeaTalk 消息，把结果（成功/失败、返回内容）返回给 Agent。
6. **Agent** 把「工具执行结果」再塞回对话，交给 **LLM**。
7. **LLM** 根据结果生成面向用户的回复，例如：「已经通过 SeaTalk 发送了 hi」。
8. **Agent** 把这段回复展示给用户。

核心点：**Agent 和 LLM 之间多了一轮「工具调用 + 工具结果」的回合**；工具由谁实现（本地还是 MCP）不影响这条逻辑链。

---

## 三、MCP 和 Agent 在流程里各站在哪一侧

- **MCP 所在的一侧**：  
  - 事先定义好「有哪些工具、每个工具叫什么、需要什么参数、返回什么」。  
  - 当收到「调用工具 X，参数是 Y」的请求时，执行具体逻辑（例如发 HTTP 请求到 SeaTalk），并把执行结果按约定格式返回。

- **Agent 所在的一侧**：  
  - **Agent** 要先把「有哪些工具可用」告诉 LLM，有两种方式：**从 MCP 发现**，或像本项目这样**手动声明**。所谓「发现」，是指 Agent 作为 MCP 客户端，向已连接的 MCP 服务发起「列出工具」的请求（协议里通常有类似 tools/list 的约定），MCP 服务返回工具清单（名字、描述、参数等），Agent 据此得知有哪些工具、怎么描述给 LLM。  
  - 在对话中，Agent 把用户的输入交给 LLM；当 LLM 说「我要调工具 send_seatalk_message，参数是 {"content":"hi"}」时，Agent 去**真正执行**这次调用（向 MCP 发调用请求，或调本地函数），再把**执行结果**作为一条消息交给 LLM，让 LLM 生成最终回复。

所以：**MCP 负责「暴露并执行工具」，Agent 负责「理解用户、决定调用、收集结果、再交给 LLM 生成回复」。** 二者通过「工具名 + 参数 → 执行 → 结果」这一套约定来交互，而不是靠技术细节耦合。

## 四、本项目的对应关系（帮助理解）

| 组件 | 在本项目中的体现 | 在交互中的角色 |
|------|------------------|----------------|
| **工具实现** | `tools/message_operations.py` 里的 `send_seatalk_message` | 真正执行「发 SeaTalk 消息」的代码 |
| **MCP 服务** | `main_mcp.py` 启动的进程 | 把上述函数暴露为 MCP 工具，供任何 MCP 客户端调用 |
| **Agent** | `main_agent.py` 里的循环逻辑 | 接收用户输入 → 调 LLM → 若 LLM 要调工具则执行（本项目是直接调同一份 `message_operations`）→ 把结果送回 LLM → 输出回复 |

当前 `main_agent.py` 并没有真正连到 MCP 服务器，而是**和 MCP 共用同一套工具实现**。这样做的效果是：**Agent 和 MCP 在「工具长什么样、怎么调」上是一致的**；若把 Agent 改成「通过 MCP 协议去发现并调用工具」，只是把「执行」从本地函数调用换成「向 MCP 发请求」，**交互原理不变**——都是「LLM 决定调什么 → Agent 执行 → 结果回传 LLM」。

---

## 五、为什么要区分 MCP 和 Agent

- **MCP 统一「能力」的写法**：同一套工具（名字、参数、含义）可以给 Cursor、给自建的 Agent、给别的 IDE 用，大家按同一份「说明书」调用，避免各写各的。
- **Agent 统一「怎么用能力」**：无论工具来自 MCP 还是本地，Agent 的流程都是「用户输入 → LLM 决策 → 执行工具 → 结果回填 LLM → 最终回复」，便于扩展更多工具或换不同的 LLM，而不改 MCP 侧的定义。

总结：**MCP 解决「能力如何被描述和调用」，Agent 解决「谁在对话中决定何时调用、如何把结果融入回复」；二者通过「工具调用 + 结果」这一层约定交互，这就是它们的交互原理。**
